## 1. 개요

### 목적

본 문서는 TodoList 웹 애플리케이션의 주요 기능을 실제 사용자 관점에서 구체적인 행동 흐름으로 정의한다. 개발팀이 기능 구현 시 사용자 경험의 맥락을 이해하고, QA 팀이 테스트 시나리오를 도출하는 기준 문서로 활용한다.

### 페르소나 정의

#### 페르소나 A — 김지수 (대학생, 23세)

- 상황: 학기 중 과제, 시험, 팀 프로젝트 일정을 한 곳에서 관리하고 싶음
- 목표: 마감일을 놓치지 않도록 할 일을 등록하고 상태를 한눈에 파악
- 특성: 처음 사용하는 서비스에 익숙해지는 과정이 짧음. 카테고리보다 날짜 중심으로 할 일을 인식
- 사용 환경: 주로 노트북, 이따금 스마트폰

#### 페르소나 B — 박민준 (직장인, 31세)

- 상황: 여러 프로젝트를 병행하며 업무 태스크를 카테고리별로 구분해 추적하고 싶음
- 목표: 오늘 진행 중인 업무와 기한이 지난 업무를 빠르게 파악하고 처리 상태를 업데이트
- 특성: 반복 사용 패턴이 명확하고, 앱의 느낌(테마)과 언어 설정을 본인 취향에 맞추는 것을 선호
- 사용 환경: 주로 데스크톱 브라우저

  이메일 + 비밀번호 + 이름 입력                                          이메일 + 비밀번호 입력
  → "기본" 카테고리 자동 생성 (BR-04)                                    → JWT 발급
      │                                                                    │ (v2) theme/language 복원
      └────────────────────────┬───────────────────────────────────────────┘
                               │
                               ▼
                    ══════════════════════
                    [인증된 사용자 세션]
                    ══════════════════════
                               │
              ┌────────────────┼────────────────┐
              │                │                │

## 3. v1 핵심 시나리오

### 시나리오 US-01: 신규 사용자 온보딩 (회원가입 → 로그인 → 첫 할 일 등록)

**관련 UC**: UC-01, UC-02, UC-04
**페르소나**: 김지수 (학생)

#### 상황

학기 초 과제 마감 일정이 쏟아지기 시작한 김지수는 TodoList 서비스를 처음 방문한다. 아직 계정이 없는 상태다.

#### 행동 흐름 — 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | `/register` 페이지에서 이메일, 비밀번호, 이름 입력 후 가입 버튼 클릭 | 계정 생성. "기본" 카테고리 자동 생성 (BR-04). 로그인 페이지로 이동 |
| 2 | `/login` 페이지에서 방금 가입한 이메일과 비밀번호 입력 후 로그인 버튼 클릭 | JWT 토큰 발급. 할 일 목록 화면(`/`)으로 이동 |
| 3 | 빈 목록 화면을 확인한 후 할 일 등록 버튼 클릭 → `/todos/new` 이동 | 할 일 등록 폼 표시 |
| 4 | 제목 "운영체제 과제 제출" 입력, 카테고리는 선택하지 않음, 종료일 "2026-06-10" 입력 후 등록 버튼 클릭 | 카테고리 미지정이므로 "기본" 카테고리 자동 배정 (BR-03). 할 일 생성 완료. 목록 화면으로 이동 |
| 5 | 목록 화면에서 방금 등록한 할 일 확인 | 상태 "시작 전"으로 표시 (종료일만 있고 시작일 없음, is_completed=false) |

#### 결과

김지수는 별도 카테고리 설정 없이도 첫 할 일을 등록하고, 마감일 기반의 상태를 즉시 확인할 수 있다.

### 시나리오 US-02: 할 일 관리 (등록 / 수정 / 삭제 / 완료 처리)

**관련 UC**: UC-04, UC-06, UC-07, UC-08
**페르소나**: 박민준 (직장인)

#### 상황

박민준은 이미 계정을 보유하고 로그인된 상태다. 오늘 새로운 업무 태스크가 생겼고, 기존 태스크 중 하나는 완료했으며, 불필요한 태스크 하나는 삭제하려 한다.

#### 행동 흐름 — 정상 흐름 (Happy Path)

**[할 일 등록]**

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | `/todos/new` 에서 제목 "API 설계 문서 작성", 카테고리 "업무", 시작일 "2026-05-27", 종료일 "2026-05-30" 입력 후 등록 | 할 일 생성. 오늘이 시작일이므로 상태 "진행 중"으로 계산됨 |

**[할 일 수정]**

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 2 | 목록에서 기존 할 일 "DB 스키마 검토" 선택 → `/todos/:id/edit` 이동 | 기존 값이 채워진 수정 폼 표시 |
| 3 | 종료일을 "2026-05-29"로 변경 후 저장 | 변경 내용 저장. 수정된 상태 재계산 후 목록 화면으로 이동 |

**[완료 처리]**

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 4 | 목록 화면에서 "API 명세 검토" 항목의 완료 버튼 클릭 | `is_completed = true` 로 변경. 해당 항목 상태 "완료"로 즉시 갱신 |

**[할 일 삭제]**

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 5 | 목록 화면에서 "임시 메모" 항목의 삭제 버튼 클릭 | 영구 삭제 처리 (BR-05). 목록에서 즉시 제거. 복구 불가 |

#### 결과

박민준은 목록 화면에서 인라인 액션만으로 완료 처리와 삭제를 빠르게 수행하고, 수정이 필요한 경우에만 수정 화면으로 이동한다.

### 시나리오 US-03: 필터링으로 할 일 현황 파악

**관련 UC**: UC-05
**페르소나**: 박민준 (직장인)

#### 상황

박민준은 월요일 오전, 이번 주 처리해야 할 업무를 파악하려 한다. 전체 할 일이 20개 이상 쌓여 있어 상태별로 필터링해서 확인한다.

#### 행동 흐름 — 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 할 일 목록(`/`) 화면 접근 | 본인 소유 전체 할 일 목록 표시. 타인 데이터 미포함 (BR-02) |
| 2 | "기한 초과" 필터 선택 | `is_completed=false AND end_date < 오늘` 조건의 항목만 표시 (BR-12) |
| 3 | 기한 초과 항목 2건 확인 후 "진행 중" 필터로 전환 | `is_completed=false AND start_date <= 오늘 AND end_date >= 오늘` 조건의 항목만 표시 (BR-10) |
| 4 | "업무" 카테고리 필터 추가 선택 | "업무" 카테고리에 속하면서 진행 중인 항목만 표시 (BR-08) |
| 5 | "완료" 필터 선택 | `is_completed=true` 인 항목만 표시 (BR-11) |

#### 결과

박민준은 필터를 조합해 기한 초과 태스크를 먼저 처리하고, 오늘 진행 중인 업무를 카테고리별로 빠르게 파악한다.

## 4. v2 시나리오

### 시나리오 US-05: 테마 설정 및 로그인 시 복원

**관련 UC**: UC-09, UC-02, UC-03
**페르소나**: 박민준 (직장인)

#### 상황

박민준은 야간 작업이 많아 다크 모드를 선호한다. 앱의 기본 테마(라이트 모드)를 다크 모드로 변경하고, 다음 로그인 시에도 유지되기를 원한다.

#### 행동 흐름 — 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 로그인 후 `/profile` (내 정보 수정) 화면 접근 | 현재 테마 설정값("라이트 모드") 표시 |
| 2 | 테마 선택 항목에서 "다크 모드" 선택 후 저장 | `theme_preference = dark` 저장 (BR-13). UI 전체가 즉시 다크 모드로 전환 |
| 3 | 로그아웃 후 다음 날 다시 로그인 | 로그인 성공 시 서버가 `theme_preference = dark` 반환 (BR-14). 클라이언트가 즉시 다크 모드 적용. 별도 설정 없이 유지됨 |

#### 결과

박민준은 한 번의 설정으로 이후 모든 세션에서 다크 모드가 자동 유지된다.

#### 행동 흐름 — 예외 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | API로 유효하지 않은 테마 값(`blue` 등) 저장 요청 | 오류 반환 (BR-13). 기존 설정 유지. 변경되지 않음 |

### 시나리오 US-06: 언어 변경 및 로그인 시 복원

**관련 UC**: UC-10, UC-02, UC-03
**페르소나**: 김지수 (학생)

#### 상황

김지수는 영어 공부를 위해 앱 UI를 영어로 변경해보고 싶다. 변경 후 로그아웃하고 다시 로그인했을 때도 영어 설정이 유지되기를 원한다.

#### 행동 흐름 — 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | 로그인 후 `/profile` 화면 접근 | 현재 언어 설정값("한국어") 표시 |
| 2 | 언어 선택 항목에서 "English" 선택 후 저장 | `language_preference = en` 저장 (BR-15). 앱 전체 UI 텍스트(메뉴, 버튼, 레이블, 오류 메시지)가 즉시 영어로 전환 |
| 3 | 로그아웃 후 다시 로그인 | 로그인 성공 시 서버가 `language_preference = en` 반환 (BR-16). 클라이언트가 즉시 영어 UI 적용 |
| 4 | 한국어로 다시 변경하고 싶어 `/profile`에서 "한국어" 선택 후 저장 | `language_preference = ko` 저장. 앱 전체가 즉시 한국어로 전환 |

#### 결과

김지수는 언어 설정을 자유롭게 전환할 수 있으며, 변경 즉시 앱 전체에 반영되고 로그인 이후에도 유지된다.

#### 행동 흐름 — 예외 흐름

| 단계 | 사용자 행동 | 시스템 응답 |
|------|------------|------------|
| 1 | API로 유효하지 않은 언어 코드(`fr` 등) 저장 요청 | 오류 반환 (BR-15). 기존 언어 설정 유지. 변경되지 않음 |
